# Build Sequence — Commander SDR Kiro Programme v1.3.1

## Doctrine

No MVP. The whole Commander SDR product journey is committed from day one. Implementation is staged by version, but domains are not discarded.

## Prerequisite chain

1. Authority and baseline reading before all work.
2. Shell, route registry and design system before UI feature implementation.
3. BP-03 canonical data model before domain features consuming canonical entities.
4. BP-13 connector framework and BP-03 before any connector pack ships.
5. Mock/seed data before Phase 2 real connector approval.
6. Security C2/OODA doctrine #56–#62 must be read before C2, intelligence, connector, verdict or surface work.
7. Strategy Layer Runtime Surface must precede case management, routing, validation/closure, reopening and Fusion Map implementation because baseline Spec #32 is mandatory and build-blocking.

## Gate 0 — Kiro pack validation

**Purpose:** Validate authority, steering, specs, build packs, prompt library and folder skeleton.

**Allowed:** Markdown edits, spec refinement, roadmap refinement, task reordering, test schedule refinement.

**Blocked:** App code, live AWS resources, real vendor connectors, production billing, n8n and custom Kiro powers.
